\section{Introduction}
%=============================================================================

Large language models (LLMs) have enabled autonomous agents capable of reasoning, tool use, and multi-turn interaction with complex environments \citep{yao2023react, jimenez2023swe, su2025learnbyinteract}. Agentic reinforcement learning (Agentic RL), where agents are trained directly from verifiable environment rewards through multi-turn interactions, has gained increasing attention as a promising paradigm for developing capable agents \citep{shao2024deepseekmath, guo2025deepseek, jin2025search, xue2025simpletir}. However, agents trained purely through Agentic RL tend to overfit to environment-specific patterns rather than acquiring general problem-solving strategies, limiting their transfer to unseen tasks and domains.

Recent work has shown that modular \textit{skills}, reusable experience snippets that encode task-specific strategies, can greatly facilitate agent problem solving \citep{wang2024awm, cao2025reme, fang2025memp}. A pioneering effort in this direction is Anthropic's Skill Creator \citep{anthropic2025skillcreator}, which automates skill creation through an iterative human-in-the-loop process: skills are written, blind-tested by an independent comparator, and refined based on user feedback (Figure~\ref{fig:intro_comparison}, left panel). A natural question arises: \textit{can we build a fully automated Skill Creator that replaces human judgment with RL reward signals, enabling skills and policy to co-evolve during training?}

\begin{figure*}[!t]
\centering
\includegraphics[width=\textwidth]{figures/intro_figure.pdf}
\caption{\textbf{Left two panels}: From human-in-the-loop to RL-in-the-loop skill creation. Anthropic's Skill Creator~\citep{anthropic2025skillcreator} iteratively writes skill versions, blind-tests them, and refines based on human feedback; \ours~replaces the human loop with the RL training loop. \textbf{Right panel}: Compared to existing skill evolution approaches (EvolveR, SkillRL, etc.) that update skills without reward-based evaluation, \ours~dynamically allocates rollouts across skill versions ($\mathcal{V}_{\text{old}}$/$\mathcal{V}_{\text{new}}$) and uses Thompson Sampling to accept or reject changes based on verifiable reward signals.}
\label{fig:intro_comparison}
\end{figure*}

Despite growing interest, there remains a significant gap in faithfully integrating the skill creation paradigm with agentic RL. Recent approaches accumulate experiential knowledge through verbal reflection \citep{shinn2023reflexion}, episodic extraction \citep{zhao2024expel}, memory evolution \citep{zhang2026memrl}, or teacher distillation \citep{xia2025skillrl}, but none adopt the key ingredient of the Skill Creator: \textit{controlled, blind comparison} of skill versions under identical conditions. These methods accept whatever skills are generated without principled evaluation, treating skill creation and policy learning as separate processes rather than considering how the evolving policy digests and reconciles with changing skills (Figure~\ref{fig:intro_comparison}, right panel). The optimal skill set is not static: as the policy improves, previously helpful skills can become redundant or even harmful, and new failure modes emerge that require different guidance \citep{soylu-etal-2024-fine, liu2025understanding}.

We propose \ours, an RL-Guided Skill Creator that directly adapts the Skill Creator paradigm for agentic RL. Inspired by the blind testing mechanism of Anthropic's Skill Creator, \ours~leverages a structural property of GRPO \citep{shao2024deepseekmath}: each training step samples a group of $G$ rollouts from the same task under the same policy. We split these rollouts \textit{within each group}, where some use the candidate skill version $\mathcal{V}_{\text{new}}$ and others use the current version $\mathcal{V}_{\text{old}}$, creating a controlled comparison analogous to blind testing where the only variable is the skill configuration. The allocation is governed by Thompson Sampling \citep{thompson1933likelihood, russo2018tutorial} with an adaptive discount that accounts for the evolving policy. Each rollout thus serves a \textbf{triple role}: (i)~policy optimization via GRPO gradients, (ii)~failure trajectory collection for the skill creator pipeline, which is faithfully adapted from Anthropic's Skill Creator with assertion-based failure diagnosis and a two-step analyst/author process for versioned skill proposals (\S\ref{sec:skill_creator}), and (iii)~bandit evaluation for principled accept/reject decisions on skill versions. The total rollout count remains identical to standard GRPO; the only additional cost is periodic LLM calls for the skill creator, which constitute negligible overhead relative to rollout generation.

We thoroughly investigate \ours~on search and embodied reasoning benchmarks across multiple model scales, and further extend it to coding, web shopping, and science discovery tasks. \ours~achieves substantial convergence speedup over GRPO and static skill baselines, with strong generalization to unseen tasks. The skill lifecycle analysis (\S\ref{sec:experiments}) shows skills being automatically created, tested, refined, and pruned as the policy improves, demonstrating seamless co-evolution with marginal computational overhead.

\section{Preliminaries}
%=============================================================================

\rparagraph{Multi-Turn Agentic RL}
We consider an LLM-based agent with policy $\pi_\theta$ that interacts with an environment over multiple turns via a tool engine $\mathcal{T}$. Given a task $q \sim \mathcal{D}$ and a set of active skills $\mathcal{S}_{\text{active}} \subseteq \mathcal{S}$ (conditional behavioral guidance; defined in \S\ref{sec:skill_creator}) loaded into the context, the agent generates a trajectory $\tau$ of interleaved actions and observations. The learning objective is to maximize the expected reward under a rule-based verifier $r(\tau) \in \{0, 1\}$:
\begin{equation}
    \max_{\pi_\theta} \mathbb{E}_{q \sim \mathcal{D},\, \tau \sim \pi_\theta(\cdot|q, \mathcal{S}_{\text{active}};\mathcal{T})} \left[ r(\tau)
    \right].
    \label{eq:rl_objective}
\end{equation}

\rparagraph{Group Relative Policy Optimization (GRPO)}
GRPO \citep{shao2024deepseekmath} optimizes the policy without a separate value function by exploiting \textit{group-wise sampling}. For each task $q \sim \mathcal{D}$, GRPO samples a group of $G$ trajectories $\{\tau_i\}_{i=1}^G$ from the current policy $\pi_{\theta}(\cdot \mid q, \mathcal{S}_{\text{active}}; \mathcal{T})$, computes a group-normalized advantage $\hat{A}_i$ from their rewards, and updates $\theta$ via a clipped surrogate objective (details in Appendix~\ref{app:grpo}). The key structural property we exploit is that all $G$ rollouts in a group share the same task input $q$ and the same policy $\pi_\theta$, differing only in their stochastic trajectories. This provides a natural, controlled setting for comparing two skill configurations within the same group (\S\ref{sec:design}).

\section{RL-Guided Skill Creation (\ours)}
\label{sec:method}
%=============================================================================

Recent memory and skill-based approaches already update auxiliary guidance during training, whether through instruction population evolution \citep{zhou2025inspo}, episodic extraction \citep{zhao2024expel}, or teacher distillation \citep{xia2025skillrl}. Yet a key idea from Anthropic's Skill Creator \citep{anthropic2025skillcreator} remains under-explored in this literature: \textit{controlled, blind testing} that ensures only genuinely helpful modifications are retained. Without such testing, skill libraries accumulate untested changes whose individual effects are never isolated, making it impossible to distinguish beneficial guidance from noise.
\ours~addresses this by integrating skill creation, evaluation, and evolution directly into the GRPO training loop.


\subsection{Within-Group Skill Comparison and Triple-Duty Rollouts}
\label{sec:design}

\rparagraph{Within-Group Splitting}
Blind testing requires comparing two skill configurations under identical conditions. In agentic RL, a natural comparison unit is the task: run one group under $\mathcal{V}_{\text{new}}$ and another under $\mathcal{V}_{\text{old}}$. However, multi-turn rollouts are expensive and group sizes are modest, so splitting across \textit{tasks} introduces high variance from task difficulty alone. We instead split across \textit{rollouts within each group}: all $G$ rollouts share the same task $q$ and the same policy $\pi_\theta$, so the only variable is the skill configuration. This is analogous to the within-subject design in controlled experiments, where each subject (task) serves as its own control.

Concretely, for each rollout $i$ in a group, a version assignment $d_i$ is drawn from the Thompson Sampling posterior (\S\ref{sec:skill_evolution}), and the rollout is generated under the corresponding skill set $\mathcal{S}_{d_i}$. The \ours~objective extends the standard GRPO formulation (Appendix~\ref{app:grpo}) as:
\begin{equation}
\begin{aligned}
\mathcal{J}_{\text{\ours}}(\theta) = \mathbb{E}_{q \sim \mathcal{D},\; \textcolor{ourblue}{d_i \sim \mathrm{TS}(\alpha, \beta)},\; \tau_i \sim \pi_{\theta_{\text{old}}}(\cdot \mid q,\, \textcolor{ourblue}{\mathcal{S}_{d_i}};\, \mathcal{T})} \left[ \mathcal{L}_{\text{GRPO}}(\theta;\, \{\tau_i\}_{i=1}^G) \right],
\end{aligned}
\label{eq:scrl_objective}
\end{equation}
where the \textcolor{ourblue}{blue} terms denote the components introduced by \ours: $\textcolor{ourblue}{d_i \sim \mathrm{TS}(\alpha, \beta)}$ assigns each rollout to either $\mathcal{V}_{\text{new}}$ or $\mathcal{V}_{\text{old}}$ via Thompson Sampling, and $\textcolor{ourblue}{\mathcal{S}_{d_i}}$ is the active skill set under the assigned version. This contrasts with approaches like INSPO \citep{zhou2025inspo}, which sample one instruction per group (all $G$ rollouts share it); \ours~instead varies the skill configuration \textit{within} the group, enabling controlled comparison at the rollout level.

This formulation means each rollout simultaneously serves three purposes without additional cost: its reward provides GRPO gradients for policy optimization (the primary purpose), completed trajectories feed an experience reservoir $\mathcal{R}$ that drives the skill creator (\S\ref{sec:skill_creator}), and its outcome under the assigned version updates the Thompson Sampling posteriors for the accept/reject decision (\S\ref{sec:skill_evolution}). The total rollout count remains identical to standard GRPO; the only overhead is periodic LLM calls for the skill creator, which constitute negligible cost relative to rollout generation.

\begin{algorithm}[t]
\begin{footnotesize}
\caption{\ours: RL-Guided Skill-Policy Co-Evolution}
\label{alg:scrl}
\begin{algorithmic}[1]
    \STATE \textbf{Input}: Policy $\pi_{\theta_0}$, training steps $T$, evolution frequency $K_e$.
    \STATE \textbf{Initialize}: Skill bank $\mathcal{S} {=} \emptyset$, reservoir $\mathcal{R} {\leftarrow} \emptyset$, posteriors $(\alpha_a, \beta_a) {\leftarrow} (1,1)$ for $a {\in} \{\text{new}, \text{old}\}$, memory $\hat{M} {\leftarrow} \infty$.
    \FOR{$t=1, \dots, T$}
        \STATE \textcolor{paleblue}{[\textbf{GRPO Rollout with Skill A/B Testing}]}
        \STATE Sample task $q \sim \mathcal{D}$. For each rollout $i$: draw $p_a \sim \text{Beta}(\alpha_a, \beta_a)$, set $d_i {=} \mathbf{1}[p_{\text{new}} {>} p_{\text{old}}]$.
        \STATE Load active skills from $\mathcal{S}_{d_i}$ into context; generate $\{\tau_i\}_{i=1}^G \sim \pi_{\theta_{t-1}}(\cdot | q, \mathcal{S}_{d_i}; \mathcal{T})$; compute $\{r_i\}$.
        \STATE Update policy: $\theta_t \leftarrow \text{GRPO}(\theta_{t-1}, \{\tau_i\}, \{r_i\})$. \hfill {\color{gray}\textit{// policy optimization}}
        \STATE Add $\{(\tau_i, d_i, r_i)\}$ to reservoir $\mathcal{R}$. \hfill {\color{gray}\textit{// experience collection}}
        \STATE $w_t {\leftarrow} \hat{M}/(\hat{M}{+}n_{t,a})$;\; $\alpha_a {\leftarrow} w_t \alpha_a {+} m_{t,a}$;\; $\beta_a {\leftarrow} w_t \beta_a {+} (n_{t,a} {-} m_{t,a})$,\; for $a {\in} \{\text{new}, \text{old}\}$. \hfill {\color{gray}\textit{// update sampler}}
        \IF{$t \bmod K_e = 0$}
            \STATE \textcolor{palered}{[\textbf{Evolution}]} Accept $\mathcal{V}_{\text{new}}$ if $\bar{\mu}_{\text{new}} {>} \bar{\mu}_{\text{old}}$ (Eq.~\ref{eq:accept}); update $\hat{M}$ via predictive-likelihood.
            \STATE \textcolor{palegreen}{[\textbf{Skill Creation}]} Grade $\mathcal{R}$ with assertions $\mathcal{A}$ (Eq.~\ref{eq:failure_profile}); diagnose failures; propose new $\mathcal{V}_{\text{new}}$.
            \STATE Reset posteriors: $(\alpha_a, \beta_a) \leftarrow (1, 1)$; begin testing $\mathcal{V}_{\text{new}}$ vs.\ $\mathcal{V}_{\text{old}}$.
        \ENDIF
    \ENDFOR
    \STATE \textbf{Return} optimized policy $\pi_{\theta_T}$ and skill bank $\mathcal{S}$.
\end{algorithmic}
\end{footnotesize}
\end{algorithm}

\subsection{Skill Creator Pipeline}
\label{sec:skill_creator}

The within-group split (\S\ref{sec:design}) provides the evaluation mechanism; this section describes the creator pipeline that produces candidate skill versions for evaluation. We adapt the diagnosis-proposal-validation loop from Anthropic's Skill Creator \citep{anthropic2025skillcreator} for fully automated use within the RL training loop.

\rparagraph{Experience Reservoir}
In the original Skill Creator, a human practitioner supplies execution examples for diagnosis. We replace this manual step with a continuously refreshed experience reservoir $\mathcal{R}$ that selectively retains trajectories from ongoing GRPO training (Eq.~\ref{eq:scrl_objective}), each annotated with its version label $d_i$ and reward $r(\tau_i)$. Because $\mathcal{R}$ evolves alongside the policy, it reflects the current behavioral landscape and provides the raw material for the diagnosis-proposal loop below.

\rparagraph{Conditional Skill Loading}
Each skill $S_k = (c_k, \text{trig}_k)$ pairs structured content $c_k$ (situational applicability, action guidance, and counterexamples) with an iteratively refined trigger condition $\text{trig}_k$ parameterized by type (Appendix~\ref{app:triggers}). At each step, only skills whose triggers fire are loaded into the context, yielding $\mathcal{S}_{\text{active}} \subseteq \mathcal{S}$ from Eq.~(\ref{eq:rl_objective}). Among triggered skills, the agent further selects based on each skill's \texttt{when\_to\_use} field: the trigger gates context loading, while the model's own judgment gates adoption. This two-level filter ensures targeted guidance without inflating every context.

\rparagraph{Assertion-Based Failure Grading}
We adopt the assertion mechanism from Anthropic's Skill Creator and extend it into a rubrics-based grading module that evaluates trajectories across the entire reservoir without repetitive LLM calls. We maintain a dynamic assertion set $\mathcal{A} = \{\phi_j\}_{j=1}^{|\mathcal{A}|}$, where each $\phi_j$ is a rule-based predicate over trajectories, e.g., validating whether the agent verified its answer before terminating. Applying every assertion uniformly across the reservoir yields an aggregate failure profile:
\begin{equation}
    \hat{r}_j = \frac{1}{|\mathcal{R}|}\sum_{\tau \in \mathcal{R}} \phi_j(\tau), \quad j = 1, \ldots, |\mathcal{A}|, \qquad \mathbf{f} = [\hat{r}_1, \ldots, \hat{r}_{|\mathcal{A}|}],
    \label{eq:failure_profile}
\end{equation}
where each $\hat{r}_j \in [0,1]$ is the pass rate of assertion $j$ across $\mathcal{R}$, and $\mathbf{f}$ is the aggregate failure profile vector. An LLM-based analyzer then receives $\mathbf{f}$ alongside a stratified sample of trajectories and serves two roles: (1)~it produces a prevalence-ranked failure diagnosis $\mathcal{F}_{\text{diag}}$ that identifies the dominant behavioral patterns, and (2)~it evolves $\mathcal{A}$ itself via add, delete, and modify operations, adapting the rubric as the policy's failure landscape shifts (Appendix~\ref{app:skill_pipeline}).

\rparagraph{History-Informed Skill Revision}
Given $\mathcal{F}_{\text{diag}}$, a skill creator agent revises the skill bank $\mathcal{S}$ by proposing a candidate version $\mathcal{V}_{\text{new}}$: a coherent set of \textsc{add}, \textsc{modify}, and \textsc{delete} operations over skills. The agent receives the current skill bank, the failure diagnosis with representative episodes, and the tracked history of prior proposals along with their accept/reject outcomes from Thompson Sampling (\S\ref{sec:skill_evolution}). Conditioning on this history enables the agent to avoid repeating rejected strategies and to build on previously accepted changes, yielding increasingly targeted revisions as training progresses. Implementation details of the internal agent decomposition are provided in Appendix~\ref{app:skill_pipeline}.

\rparagraph{Trigger Validation}
Before a proposed version enters A/B testing, the pipeline estimates each trigger's firing rate over reservoir episodes; proposals falling below a minimum activation threshold are rejected and rewritten, ensuring sufficient coverage for meaningful evaluation (Appendix~\ref{app:triggers}).

The creator proposes a candidate version $\mathcal{V}_{\text{new}}$; it remains to determine how rollouts are allocated between $\mathcal{V}_{\text{new}}$ and $\mathcal{V}_{\text{old}}$, and when to accept or reject.


\subsection{RL-Guided Skill Evolution with Thompson Sampling}
\label{sec:skill_evolution}

In Anthropic's Skill Creator, blind testing is performed by launching parallel agents with and without a candidate skill on a single task. Our setting differs in two respects. First, evaluation spans multiple training steps rather than a single run: across each evolution cycle of $K_e$ steps, both $\mathcal{V}_{\text{new}}$ and $\mathcal{V}_{\text{old}}$ are tested repeatedly under the within-group split (\S\ref{sec:design}). Second, every rollout also serves policy optimization, so we seek to maximize training efficiency by allocating more rollouts to the more effective version while preserving sufficient exploration for a reliable accept/reject decision. This is a natural two-armed bandit problem, and we adopt Thompson Sampling \citep{thompson1933likelihood, russo2018tutorial} as the allocation mechanism.

\rparagraph{Version Assignment}
Each arm maintains a Beta posterior over its success probability, initialized with uniform priors $\text{Beta}(1, 1)$. At the start of each rollout, the version assignment $d_i$ from Eq.~(\ref{eq:scrl_objective}) is sampled by drawing $p_{\text{new}} \sim \text{Beta}(\alpha_{\text{new}}, \beta_{\text{new}})$ and $p_{\text{old}} \sim \text{Beta}(\alpha_{\text{old}}, \beta_{\text{old}})$, then assigning:
\begin{equation}
    d_i = \mathbf{1}[p_{\text{new}} > p_{\text{old}}], \quad \text{with } \Pr(\text{new}) = \text{clamp}\!\left(\Pr(p_{\text{new}} > p_{\text{old}}),\, \varepsilon_{\text{ts}},\, 1{-}\varepsilon_{\text{ts}}\right),
    \label{eq:thompson_decision}
\end{equation}
where $\varepsilon_{\text{ts}}$ ensures sufficient exploration of both versions. All steps within a single rollout use the assigned version's skill library.

\rparagraph{Sample-Size-Adaptive Discount}
Unlike a classical bandit where arm rewards are stationary, the policy $\pi_\theta$ evolves at every GRPO update, rendering earlier observations increasingly stale. The posterior must discount old evidence at a rate that reflects the degree of policy change, or accumulated pseudo-counts will anchor the estimate to outdated behavior. We introduce a \textit{sample-size-adaptive discount}: at each step~$t$, existing pseudo-counts are shrunk by a factor $w_t$ before incorporating the $m_t$ successes out of $n_t$ new episodes:
\begin{equation}
    \alpha \leftarrow w_t \cdot \alpha + m_t, \quad \beta \leftarrow w_t \cdot \beta + (n_t - m_t), \qquad w_t = \frac{1}{1 + n_t/M},
    \label{eq:beta_update}
\end{equation}
where $M > 0$ is a scalar \emph{memory parameter}. The adaptive form of $w_t$ is motivated by the fact that the two arms receive different numbers of samples at each step due to the evolving Thompson Sampling allocation: an arm with fewer observations ($n_t \ll M$) is more susceptible to noise and should retain more prior evidence ($w_t \to 1$), while an arm receiving many observations ($n_t \gg M$) carries sufficient signal to aggressively discount the past ($w_t \to 0$). The posterior remains a valid Beta distribution throughout, so Thompson Sampling applies directly.

\rparagraph{Memory Parameter Estimation}
The memory parameter $M$ controls how quickly the posterior forgets, but its optimal value depends on the rate of policy change, which varies across training. Rather than fixing $M$ as a hyperparameter, we estimate it from data via sequential predictive-likelihood maximization over completed evaluation cycles (Appendix~\ref{app:staleness_details}). At each cycle boundary, the system selects the $M$ that best explains the observed reward sequence under the discounted Beta model, yielding an adaptive forgetting rate that requires no manual tuning.

\rparagraph{Accept/Reject Decision}
At the end of each cycle, the candidate is accepted if its discounted posterior mean exceeds that of the current version:
\begin{equation}
    \text{accept}(\mathcal{V}_{\text{new}}) = \mathbf{1}\!\left[\bar{\mu}_{\text{new}} > \bar{\mu}_{\text{old}}\right],
    \label{eq:accept}
\end{equation}
where $\bar{\mu}_a = \mathbb{E}[p_a]$ under the discounted Beta posterior from Eq.~(\ref{eq:beta_update}). If accepted, $\mathcal{V}_{\text{new}}$ becomes the new baseline; otherwise the skill bank reverts to $\mathcal{V}_{\text{old}}$, and the next evolution cycle begins immediately.


The complete procedure is summarized in Algorithm~\ref{alg:scrl}. The skill creator replaces ad-hoc prompt engineering with structured failure analysis, while version-level Thompson Sampling provides principled, data-efficient validation of skill changes using only the rollouts already generated by GRPO training.

\subsection{Setup}

\sparagraph{Models and benchmarks}
We evaluate \ours~on two diverse agentic benchmarks: multi-hop question answering with a search tool (NQ + HotpotQA) and ALFWorld \citep{shridharalfworld} (embodied household tasks). We conduct experiments using the Qwen3 model family \citep{bai2023qwen} at two scales: 4B and 8B parameters. Unless otherwise specified, all results use the 4B model.

\rparagraph{Baselines}
We compare \ours~against two categories of baselines. \textit{Prompt-based methods}: zero-shot \textbf{Baseline}, and retrieval-augmented variants including \textbf{Retrieval} (BM25), \textbf{RAG} (HippoRAG), and \textbf{Long Context} (full trajectory history in context). \textit{RL methods}: \textbf{GRPO}, standard GRPO training with a static instruction and no skills, and \textbf{SkillRL} \citep{xia2025skillrl}, which distills teacher-model skills as static augmentations. We also evaluate ablation variants of \ours~(see \S\ref{sec:ablation}). Implementation details are provided in Appendix~\ref{app:implementation}.

\section{Related Work}
\label{sec:related}
%=============================================================================

\sparagraph{Reinforcement Learning for LLMs}
Reinforcement learning has become a powerful paradigm for post-training LLMs, from alignment with human preferences \citep{ouyang2022training, rafailov2023direct} to reasoning via verifiable rewards. DeepSeek-R1 \citep{guo2025deepseek} demonstrated the effectiveness of RLVR, where GRPO \citep{shao2024deepseekmath} enables group-wise advantage estimation without a critic model. DAPO \citep{yu2025dapo} introduces training stability improvements such as clip-higher and dynamic sampling. Dr.~GRPO \citep{liu2025understanding} rectifies length bias. \ours~is orthogonal to these algorithmic improvements and can be integrated with any GRPO-based training pipeline.

\rparagraph{Tool-Integrated Agents}
LLM agents interact with environments through tool use \citep{yao2023react}. IRCoT \citep{trivedi-etal-2023-interleaving} interleaves chain-of-thought reasoning \citep{wei2022chain} with information retrieval. Toolformer \citep{schick2023toolformer} teaches tool usage via SFT. The RL paradigm has enabled multi-turn tool interactions \citep{feng2025retool, li2025torl}, with search tools excelling at question answering \citep{jin2025search, song2025r1}. End-to-end RL frameworks for tool use \citep{xue2025simpletir, jiang2025verltool} have grown rapidly, yet the role of \textit{dynamic skill guidance} during RL training has been largely unexplored: all existing approaches rely on static instructions designed before training begins.

\rparagraph{Skill and Instruction Optimization}
The performance of LLM agents is highly sensitive to their guiding instructions \citep{zhou2025multi}. Automated prompt optimization (APO) approaches use paraphrasing \citep{zhou2023large}, LLM-based optimizers \citep{yang2024large}, or textual gradients \citep{pryzant-etal-2023-automatic, 2025textgrad} to improve instructions, but operate offline. In the skill domain, Reflexion \citep{shinn2023reflexion} accumulates verbal feedback, ExpeL \citep{zhao2024expel} extracts experiential rules, MemRL \citep{zhang2026memrl} evolves episodic memory, and SkillRL \citep{xia2025skillrl} distills teacher-model skills for RL augmentation. Anthropic's Skill Creator \citep{anthropic2025skillcreator} pioneered the automated creation of modular, trigger-based skills from agent experience. More recent concurrent work integrates guidance optimization into the RL loop: INSPO \citep{zhou2025inspo} maintains a population of instructions and samples one per group via softmax-weighted selection, while EMPO$^2$ \citep{liu2026exploratory} alternates between memory-augmented and memory-free prompting across training iterations. Both approaches, however, assign a single prompt configuration to all rollouts within a group, so the effect of a guidance change is confounded with inter-task variance. \ours~instead splits rollouts \textit{within} each group for the same task, isolating the skill version as the sole variable and enabling controlled, per-version evaluation via Thompson Sampling.

\rparagraph{Bandit Methods for Online Selection}
Thompson Sampling \citep{thompson1933likelihood} is a Bayesian approach to the multi-armed bandit problem that maintains posterior distributions over arm rewards and samples actions proportionally to their probability of being optimal \citep{russo2018tutorial}. In the context of RL-based agent training, INSPO \citep{zhou2025inspo} uses softmax-weighted sampling over an instruction population. \ours~instead employs per-skill Thompson Sampling with trigger-gated counting: each skill maintains independent Beta posteriors updated only from episodes where its trigger fires, providing a natural handle for the heterogeneous activation patterns of different skill types.

\section{Conclusion}
\label{sec:conclusion}
%=============================================================================

We introduced \ours, a framework that enables the co-evolution of modular skills and policy learning for agentic RL. By integrating an assertion-driven skill creator pipeline with Thompson Sampling-based skill evolution directly into the GRPO training loop, \ours~achieves substantial convergence speedup with zero additional rollouts. The triple-duty design, where each GRPO rollout simultaneously provides policy gradients, failure data for skill creation, and bandit evaluation signals for skill evolution, demonstrates that significant efficiency gains are possible by extracting more value from existing training computation.

\section{GRPO Objective}
\label{app:grpo}

For completeness, the full GRPO objective \citep{shao2024deepseekmath} is:
\begin{equation}
\begin{aligned}
&\mathcal{J}_{\text{GRPO}}(\theta) = \mathbb{E}_{\tau \sim \pi_{\text{old}}} \Bigg[ \frac{1}{G} \sum_{i=1}^G \frac{1}{\sum_t \mathcal{I}_{i,t}} \sum_{t=1}^{|\tau_i|}\mathcal{I}_{i,t} \cdot \min \left( \rho_{i,t}\hat{A}_{i}, \text{clip}(\rho_{i,t}, 1{-}\epsilon, 1{+}\epsilon) \hat{A}_{i} \right)  - \beta \mathbb{D}_{\text{KL}} \Bigg],
\end{aligned}
\label{eq:grpo}
\end{equation}
where $\rho_{i,t}(\theta) = {\pi_\theta(\tau_{i,t}|\tau_{i,<t})}/{\pi_{\theta_{\text{old}}}(\tau_{i,t}|\tau_{i,<t})}$ is the token-level importance ratio, $\hat{A}_i = ({r_i - \text{mean}(\{r_j\}_{j=1}^{G})})/{\text{std}(\{r_j\}_{j=1}^{G})}$ is the group-normalized advantage, $\epsilon$ is the clip ratio, and $\beta$ controls the KL divergence penalty $\mathbb{D}_{\text{KL}}(\pi_\theta \| \pi_{\text{ref}})$. The indicator $\mathcal{I}_{i,t}$ masks out environment observation tokens so that gradients are computed only over the agent's own tokens \citep{jin2025search, xue2025simpletir}.

\section{Skill Creator Pipeline: Full Details}
\label{app:skill_pipeline}

This appendix provides the full details of the skill creator pipeline summarized in \S\ref{sec:skill_creator}. We describe each stage in the order it executes within an evolution cycle, mapping each to the corresponding phase of Anthropic's Skill Creator \citep{anthropic2025skillcreator}.

\subsection{Reservoir Collection (Test Execution)}

In Skill Creator, each evaluation cycle spawns parallel subagents to execute a task with and without the skill, producing paired outputs for comparison. In \ours, the within-group GRPO rollouts from \S\ref{sec:design} serve this role directly. Each trajectory $\tau_i$ is already tagged with its version assignment $d_i \in \{\mathcal{V}_{\text{new}}, \mathcal{V}_{\text{old}}\}$ and binary reward $r(\tau_i)$. Failed trajectories ($r(\tau_i) < 1$) enter the bounded failure reservoir $\mathcal{R}$, maintained as a sliding window over recent training steps so that the failure landscape reflects the current policy rather than historical behavior. Successful trajectories from both versions provide contrast when the analyzer selects representative episodes.

The reservoir replaces Skill Creator's small test case set (typically 2--3 prompts each run 3 times) with hundreds of diverse episodes collected at zero additional cost, since the rollouts are generated for GRPO regardless.

\subsection{Assertion Grading and Reservoir-Level Diagnosis}

\rparagraph{Assertion Types}
The assertion grader $\mathcal{G}$ maintains a dynamic set of rule-based assertions $\mathcal{A} = \{\phi_j\}$, where each assertion $\phi_j = (\text{check}_j, \text{params}_j)$ is a predicate over trajectories. The following assertion types are supported:

\begin{itemize}[itemsep=2pt, leftmargin=*]
    \item \texttt{action\_exists}: Tests whether a specific action (by regex) appears anywhere in the trajectory. Example: $\text{params} = \{\texttt{pattern}: \texttt{"search\textbackslash[.*\textbackslash]"}\}$ checks whether the agent ever issued a search action.
    \item \texttt{action\_not\_exists}: Tests whether an action pattern is \textit{absent} from the trajectory.
    \item \texttt{action\_before}: Tests temporal ordering between two action patterns. Example: $\text{params} = \{\texttt{first}: \texttt{"search"}, \texttt{second}: \texttt{"submit"}\}$ checks that a search preceded the final submission.
    \item \texttt{final\_action}: Tests whether the last action matches a given pattern. Useful for detecting premature termination.
    \item \texttt{no\_consecutive\_repeat}: Tests whether the agent avoids repeating the same action consecutively more than $n$ times.
    \item \texttt{min\_steps} / \texttt{max\_steps}: Tests trajectory length bounds. Detects both under-exploration and unproductive looping.
    \item \texttt{action\_count}: Tests whether a specific action type appears at least (or at most) $n$ times.
    \item \texttt{observation\_contains}: Tests whether the environment's observation at any step contains a target string, checking whether the agent reached specific states.
\end{itemize}

\rparagraph{Grading}
Given the failure reservoir $\mathcal{R}$, the grader evaluates every trajectory against all assertions, producing per-assertion pass rates:
\begin{equation}
    \hat{r}_j = |\mathcal{R}|^{-1} \sum_{\tau \in \mathcal{R}} \mathbf{1}[\text{check}_j(\tau, \text{params}_j)].
\end{equation}
Low pass rates identify prevalent failure modes (e.g., $\hat{r}_j = 0.23$ for \texttt{action\_before(search, submit)} reveals that 77\% of failures never searched before submitting). The grading is entirely rule-based with no LLM calls, enabling evaluation of the full reservoir rather than a subsample.

\rparagraph{Assertion Analysis and Evolution}
An LLM-based assertion analyzer $\mathcal{Z}$ receives: (1) the per-assertion pass rates, (2) a stratified sample of trajectories from $\mathcal{R}$ (balancing successes and failures, diversity of failure types), and (3) a summary of the current skill bank $\mathcal{S}$. The analyzer produces two outputs:

\begin{enumerate}[itemsep=2pt]
    \item \textbf{Failure diagnosis} $\mathcal{F}_{\text{diag}}$: A prevalence-ranked list of failure patterns, each annotated with the supporting assertion pass rates and selected representative episodes. The diagnosis identifies which behavioral patterns most need skill intervention.
    \item \textbf{Assertion CRUD operations}: The analyzer evolves $\mathcal{A}$ itself:
    \begin{itemize}[itemsep=1pt]
        \item \textsc{add}: Introduces new assertions for failure modes not covered by existing checks (e.g., discovering that the agent repeatedly visits already-explored locations, motivating a new \texttt{no\_consecutive\_repeat} assertion).
        \item \textsc{delete}: Removes non-discriminating assertions that always pass regardless of trajectory quality, analogous to Skill Creator's grader flagging ``weak assertions.''
        \item \textsc{modify}: Refines assertion parameters (e.g., tightening a regex pattern or adjusting a step count threshold) as the policy's behavior evolves.
    \end{itemize}
\end{enumerate}

This self-evolving assertion mechanism mirrors Skill Creator's grader self-critique, where the Grader agent flags non-discriminating assertions and missing coverage. The key adaptation is scale: Skill Creator runs assertions against 2--3 test cases; we evaluate across the entire reservoir, giving the assertion pass rates statistical significance.

\subsection{Version Proposal: Analyst--Author Decomposition}

The main text presents version proposal as a single agent call. In implementation, it is decomposed into two LLM calls with distinct roles, following the principle of separating analytical reasoning from creative content generation.

\rparagraph{Version Analyst}
The version analyst receives:
\begin{itemize}[itemsep=2pt, leftmargin=*]
    \item The failure diagnosis $\mathcal{F}_{\text{diag}}$ with prevalence-ranked patterns
    \item The current skill bank $\mathcal{S}$ with all active skills
    \item The complete history of prior version proposals and their accept/reject outcomes from Thompson Sampling, including the A/B success rates that led to each decision
    \item Action-pattern frequency statistics from recent rollouts (which patterns the agent commonly produces)
    \item Selected representative episodes (both failures illustrating the diagnosed patterns and successes for contrast)
\end{itemize}

The analyst produces a \textit{change plan} $\mathcal{P}$: a coherent set of intended operations, each specifying:
\begin{itemize}[itemsep=2pt, leftmargin=*]
    \item \textbf{Operation type}: \textsc{add} (new skill), \textsc{modify} (revise existing skill), or \textsc{delete} (remove skill)
    \item \textbf{Rationale}: Which failure pattern from $\mathcal{F}_{\text{diag}}$ motivates this operation
    \item \textbf{Intent}: High-level description of what the skill should achieve (for \textsc{add}/\textsc{modify})
    \item \textbf{Target}: Which existing skill to modify or delete (for \textsc{modify}/\textsc{delete})
\end{itemize}

The analyst prioritizes: unaddressed failure patterns for additions; skills whose guidance contradicts observed successful behavior for modifications; and skills the policy has internalized (high reward with or without the skill, or chronically low trigger rates) for deletions. The total number of active skills is capped at $K_{\max}$.

\rparagraph{Version Author}
The version author receives the change plan $\mathcal{P}$ along with a scoped context: only the referenced existing skills and the trajectories cited in the plan's rationale. This scoping prevents the author from being distracted by the full diagnostic context and focuses it on executing the plan.

For each operation in $\mathcal{P}$, the author writes:
\begin{itemize}[itemsep=2pt, leftmargin=*]
    \item \textbf{Skill content} $c_k$: Three structured fields following the SKILL.md format:
    \begin{itemize}[itemsep=1pt]
        \item \texttt{when\_to\_use}: Situational applicability description (when the agent should consider this skill)
        \item \texttt{action}: Concrete behavioral guidance
        \item \texttt{examples}: DO/DON'T trajectory snippets grounded in the representative episodes
    \end{itemize}
    \item \textbf{Trigger condition} $\text{trig}_k$: Type (general, beginning, or action-pattern) and pattern (regex for action-pattern triggers)
    \item \textbf{Metadata}: Skill name, description, and version number
\end{itemize}

The combined output of analyst and author constitutes the candidate version $\mathcal{V}_{\text{new}}$, which then enters the verification gate before A/B testing.

\rparagraph{Version History as Cross-Cycle Learning}
In Skill Creator, human-written \texttt{feedback.json} gates every body improvement: the human reviews outputs in the eval-viewer, writes specific feedback, and Claude rewrites accordingly. In \ours, the version history replaces this feedback loop. Each completed A/B test records: the proposed operations, the per-version success rates under Thompson Sampling, and the accept/reject outcome. The analyst sees this full history when proposing the next version, enabling it to: avoid repeating rejected strategies, build incrementally on accepted changes, and calibrate the aggressiveness of proposed modifications based on past acceptance rates.

\section{Trigger Condition Types and Validation}
\label{app:triggers}

\subsection{Trigger Types}

Each skill $S_k = (c_k, \text{trig}_k)$ has a deterministic trigger condition evaluated at every step of a rollout. The trigger determines whether the skill's content is loaded into the agent's context at that step.

\rparagraph{General Triggers}
Skills with \texttt{trigger\_type = general} fire at every step of every episode. These encode broad strategic principles applicable regardless of the agent's current state (e.g., ``verify the answer before submitting''). General skills are always present in the agent's context.

\rparagraph{Beginning Triggers}
Skills with \texttt{trigger\_type = beginning} fire only at step $t = 0$ (episode initialization). These provide planning-level guidance for how to approach the task (e.g., ``break multi-hop questions into sub-questions before searching'') without cluttering mid-episode prompts.

\rparagraph{Action-Pattern Triggers}
Skills with \texttt{trigger\_type = action\_pattern} fire when the agent's previous action $a_{t-1}$ matches a regex pattern stored in \texttt{trigger\_pattern}. For example, a skill targeting search refinement might use \texttt{trigger\_pattern = "search\textbackslash[.*\textbackslash]"} to activate only after the agent has just performed a search action. This enables context-sensitive guidance: a skill about interpreting search results only appears after a search, not during unrelated actions.

The trigger evaluation is fully deterministic and requires no LLM calls:
\begin{equation}
    \text{trig}_k(t, a_{t-1}) = \begin{cases}
        1 & \text{if type}_k = \texttt{general} \\
        \mathbf{1}[t = 0] & \text{if type}_k = \texttt{beginning} \\
        \mathbf{1}[\texttt{regex}(\texttt{pattern}_k, a_{t-1})] & \text{if type}_k = \texttt{action\_pattern}
    \end{cases}
\end{equation}

At step $t$, the active skill set is $\mathcal{S}_{\text{active}}(t) = \{S_k \in \mathcal{S} : \text{trig}_k(t, a_{t-1}) = 1\}$.

\subsection{Selective Loading via \texttt{when\_to\_use}}

Even after trigger-based filtering, multiple skills may be active simultaneously. Each skill's content $c_k$ includes a \texttt{when\_to\_use} field describing its situational applicability in natural language. The agent reads these descriptions and selects only the most relevant to its current state, creating a two-level filter:

\begin{enumerate}[itemsep=2pt]
    \item \textbf{Deterministic trigger}: Gates whether the skill's content enters the prompt at all. Low computational cost, no LLM overhead.
    \item \textbf{Model-level selection}: The agent's own judgment determines whether to \textit{adopt} the skill's guidance based on the \texttt{when\_to\_use} description and the current task context.
\end{enumerate}

This design mirrors Skill Creator's progressive disclosure architecture, where skill descriptions (~100 words) are always in context for routing, but the full skill body is loaded only when triggered. In our setting, the trigger condition serves the routing role, and \texttt{when\_to\_use} serves the relevance-filtering role.

\subsection{Trigger Validation and Optimization}

Proposed triggers must be validated before the version enters A/B testing, to ensure that skills will actually fire during evaluation.

\rparagraph{Trigger Rate Estimation}
For each proposed skill in $\mathcal{V}_{\text{new}}$ with an action-pattern trigger, the pipeline estimates the trigger's firing rate over the reservoir:
\begin{equation}
    \hat{p}_{\text{trigger}} = |\mathcal{R}|^{-1} \sum_{\tau \in \mathcal{R}} \mathbf{1}\!\left[\exists\, t : \texttt{regex}(\texttt{pattern}_k, a_t^{(\tau)}) = \text{true}\right],
\end{equation}
where $a_t^{(\tau)}$ is the action at step $t$ of trajectory $\tau$. This estimates the fraction of episodes in which the skill would activate at least once.

\rparagraph{Validation Gate}
A proposed skill passes validation if $\hat{p}_{\text{trigger}} \geq \delta_{\text{trig}}$ (default $\delta_{\text{trig}} = 0.5$). If any proposed skill in $\mathcal{V}_{\text{new}}$ fails this gate, the entire version proposal is rejected and sent back to the version author for rewriting, with the low trigger rate provided as feedback. The retry loop allows up to 3 attempts before the evolution cycle is skipped entirely.

Skills with \texttt{general} or \texttt{beginning} triggers bypass this validation, as they fire unconditionally or at a known rate.

\rparagraph{Motivation}
This validation mirrors Skill Creator's description optimization phase, which iteratively refines skill descriptions for trigger accuracy by testing against 20 evaluation queries. In our setting, the reservoir serves as the evaluation set, and the validation is a single deterministic check rather than an iterative optimization loop. A skill that never fires cannot be meaningfully evaluated in the A/B test and wastes the evaluation budget on a version that behaves identically to $\mathcal{V}_{\text{old}}$ for most episodes.

\section{Adaptive Staleness Correction: Details}
\label{app:staleness_details}

This appendix motivates the adaptive discount rule $w_t = 1/(1 + n_t/M)$ from Eq.~(\ref{eq:beta_update}), grounds it in the Bayesian posterior update, and provides the full procedure for estimating~$M$ from data.

\subsection{Motivation: From Bayesian Updates to Adaptive Discounting}

We begin from the standard Beta-Bernoulli posterior update and show how the discount rule emerges naturally from the desire to control the influence of new observations relative to accumulated evidence.

\rparagraph{The Posterior Mean as a Weighted Average}
After observing $m_t$ successes in $n_t$ episodes at step~$t$, the standard (undiscounted) Beta posterior update $\alpha_t = \alpha_{t-1} + m_t$ yields a posterior mean that is a weighted average of the prior estimate and the new observation:
\begin{equation}
    \mu_t = \frac{S_{t-1}}{S_{t-1} + n_t}\;\mu_{t-1} + \frac{n_t}{S_{t-1} + n_t}\;\frac{m_t}{n_t},
    \label{eq:standard_update}
\end{equation}
where $S_{t-1} = \alpha_{t-1} + \beta_{t-1}$ is the total pseudo-count. The weight on new evidence is $n_t/(S_{t-1} + n_t)$: the more data observed so far ($S_{t-1}$ large), the less each new batch shifts the estimate. In a stationary setting, this is optimal---the posterior concentrates as evidence accumulates.

\rparagraph{The Problem Under Non-Stationarity}
When the policy evolves, the success probability $p_{a,t}$ drifts across steps. The accumulated pseudo-count $S_{t-1}$ reflects all past policies, not just the current one. A large $S_{t-1}$ (from many past observations) anchors the estimate to stale data, making it slow to respond to genuine changes in the success rate. The fundamental issue is that the posterior treats all evidence as equally valid, regardless of when it was collected.

\rparagraph{Discounting as Effective Sample Size Control}
A geometric discount $\alpha_t = w \cdot \alpha_{t-1} + m_t$ shrinks the accumulated pseudo-count before each update, reducing the effective sample size of old evidence. With discount factor~$w$, the posterior mean becomes:
\begin{equation}
    \mu_t = \frac{w \cdot S_{t-1}}{w \cdot S_{t-1} + n_t}\;\mu_{t-1} + \frac{n_t}{w \cdot S_{t-1} + n_t}\;\frac{m_t}{n_t}.
    \label{eq:discounted_update}
\end{equation}
The key quantity is the ratio of old-to-new influence: $w \cdot S_{t-1} : n_t$. With a constant~$w$, this ratio depends on the entire allocation history through~$S_{t-1}$. In our setting, Thompson Sampling causes the per-arm episode count $n_t$ to vary by 5--10$\times$ across steps (e.g., 168 to 1112 in Search). The same constant~$w$ cannot be appropriate for both: after a step where the arm received 168 episodes, the accumulated estimate is noisy and should be retained; after a step with 1112 episodes, the new observation is precise and old evidence should be discounted more aggressively.

\rparagraph{The $M$-Parameterization}
To decouple the discount from the allocation history, we set $w_t = M/(M + n_t) = 1/(1 + n_t/M)$. Substituting into Eq.~(\ref{eq:discounted_update}), the effective old pseudo-count after discounting becomes $w_t \cdot S_{t-1} = M \cdot S_{t-1}/(M + n_t)$. This yields an intuitive interpretation:
\begin{itemize}[itemsep=2pt, leftmargin=*]
    \item When $n_t = M$ (the crossover): $w_t = 1/2$, and old and new evidence contribute equally.
    \item When $n_t \ll M$ (minority arm, noisy observation): $w_t \approx 1$, the accumulated estimate is mostly retained.
    \item When $n_t \gg M$ (majority arm, precise observation): $w_t \approx 0$, old evidence is aggressively discounted.
\end{itemize}
The parameter $M$ can be interpreted as the \emph{number of new observations at which old evidence loses half its weight}. It controls the balance between responsiveness to drift and robustness to noise, from a single scalar.

\rparagraph{Connection to Exponential Smoothing}
The update $\alpha_t = w_t \cdot \alpha_{t-1} + m_t$ with $w_t = M/(M + n_t)$ can be rewritten as
\begin{equation}
    \alpha_t = \frac{M}{M + n_t} \cdot \alpha_{t-1} + m_t = \frac{M \cdot \alpha_{t-1} + m_t \cdot (M + n_t) / 1}{M + n_t}.
\end{equation}
When $n_t = n$ is constant, this reduces to a standard geometric discount $w = M/(M+n)$, recovering exponential smoothing with a fixed forgetting rate. The generalization to variable $n_t$ preserves the exponential smoothing structure while adapting the forgetting rate to the informativeness of each observation.

\rparagraph{Why Not Estimate $w$ Directly?}
An alternative is to estimate a constant discount factor~$w^*$ by minimizing a worst-case MSE expression derived from per-step KL divergence bounds (as in importance-weighted approaches). This requires estimating the drift rate, assumes constant~$n$, and introduces a scaling parameter to handle the looseness of Pinsker's inequality. The $M$-parameterization avoids these issues: it handles variable~$n_t$ naturally and is estimated purely from observational data via the predictive likelihood described below.

\subsection{Predictive-Likelihood Criterion}

Suppose we have $K$ completed A/B tests, each spanning $T$ training steps. Test~$k$ provides per-step per-arm data $\{(m_t^{(k,a)}, n_t^{(k,a)})\}_{t=1}^T$ for each arm $a \in \{A, B\}$, where $m_t$ is the number of successes and $n_t$ is the number of episodes assigned to arm~$a$ at step~$t$.

For a candidate $M$ and one arm's data sequence within one test, the \textbf{sequential predictive log-likelihood} is:
\begin{enumerate}[itemsep=2pt]
\item Initialize $\alpha = 1$, $\beta_p = 1$ \quad ($\mathrm{Beta}(1,1)$ prior).
\item For $t = 1, \ldots, T$:
  \begin{enumerate}
    \item Discount: $\tilde\alpha = \frac{M}{M + n_t} \cdot \alpha$, \; $\tilde\beta = \frac{M}{M + n_t} \cdot \beta_p$.
    \item Score: $\ell_t = \log P\bigl(m_t \mid \mathrm{BetaBin}(n_t, \tilde\alpha, \tilde\beta)\bigr)$.
    \item Update: $\alpha = \tilde\alpha + m_t$, \; $\beta_p = \tilde\beta + (n_t - m_t)$.
  \end{enumerate}
\item Return $\sum_{t=1}^T \ell_t$.
\end{enumerate}
The BetaBinomial predictive probability at step~2(b) is $P(m \mid n, \tilde\alpha, \tilde\beta) = \binom{n}{m} B(m + \tilde\alpha, n - m + \tilde\beta) / B(\tilde\alpha, \tilde\beta)$, where $B(\cdot, \cdot)$ is the Beta function. The total log-likelihood is
\begin{equation}
    L(M) = \sum_{k=1}^K \sum_{a \in \{A, B\}} \mathrm{score}\bigl(M;\, \{m_t^{(k,a)}, n_t^{(k,a)}\}_{t=1}^T\bigr),
    \label{eq:total_ll}
\end{equation}
and the MLE is $\hat{M} = \arg\max_M L(M)$, found by grid search over $M \in [1, 2000]$.

\subsection{Online Estimation}

The log-likelihood~(\ref{eq:total_ll}) decomposes additively across tests. After each completed test~$k$, we update $L_k(M) = L_{k-1}(M) + \mathrm{score}(M;\, \text{test}~k, A) + \mathrm{score}(M;\, \text{test}~k, B)$ and re-compute $\hat{M}_k = \arg\max_M L_k(M)$. This is $O(|\text{grid}| \times T)$ work per new test, with no need to re-score history.

\subsection{Deployment Matching}

The scoring procedure mirrors the actual Thompson Sampling deployment:
\begin{itemize}[itemsep=2pt, leftmargin=*]
    \item The prior resets at the start of each test (as in deployment).
    \item The discount $w_t = M/(M + n_t)$ uses the \emph{actual} per-arm allocation at each step, not an average or fixed~$n$.
    \item The BetaBinomial score evaluates one-step-ahead prediction accuracy---the same quantity that determines posterior quality during the test.
\end{itemize}
This avoids the model misspecification that arises from estimating~$M$ using the overall success rate with a fixed per-arm count. In our experiments, such a ``naive'' approach overestimates $M$ by 1.5--2.2$\times$ because the overall rate smooths out per-arm volatility, causing the MLE to infer slower drift than actually exists.

\subsection{Empirical Validation}

We validate the estimation procedure on two domains from our experiments: Search (59 tests, per-arm $n_t \in [168, 1112]$) and ALFWorld (20 tests, per-arm $n_t \in [13, 115]$).
